# Real-Time Bridge Monitoring Platform

## Project Overview

This project demonstrates a production-minded, end-to-end real-time data engineering platform built with **Azure Databricks**, **Delta Live Tables (DLT)**, **Microsoft Fabric**, and **Power BI**.

The solution simulates IoT sensor data from five major European bridges, continuously ingesting temperature, vibration, and tilt-angle measurements through a streaming architecture. The data is validated, transformed through a Medallion Architecture (Landing → Bronze → Silver → Gold), and published to Microsoft Fabric where it is modeled for business intelligence and analytics.

The project was designed to demonstrate modern data engineering practices commonly used in enterprise environments, including real-time streaming, declarative data pipelines, data quality validation, dimensional modeling, and semantic reporting.

# Business Scenario

Bridge infrastructure is continuously monitored using IoT sensors that generate high-frequency measurements.

This project simulates three independent sensor streams:

* Temperature
* Structural vibration
* Bridge tilt angle

Streaming data is generated for five major European bridges:

* Tower Bridge (London)
* Charles Bridge (Prague)
* Rialto Bridge (Venice)
* Erasmus Bridge (Rotterdam)
* Vasco da Gama Bridge (Lisbon)

As sensor readings arrive, the platform ingests, validates, enriches, and prepares the data for downstream reporting and analytics.

Although this project focuses on bridge monitoring, the architecture is intentionally designed so that the streaming sources can be replaced with manufacturing, transportation, utilities, industrial IoT, or other operational telemetry without redesigning the platform.

# Medallion Architecture

## Landing

Stores the raw streaming Delta files exactly as generated by the simulated IoT sensors.

Purpose:

* Preserve original data
* Support replay if necessary
* Decouple ingestion from transformation

---

## Bronze

The Bronze layer performs raw ingestion into Delta Live Tables.

Responsibilities include:

* Reading streaming Delta data
* Preserving all incoming records
* Minimal transformation
* Creating an immutable historical record

---

## Silver

The Silver layer performs data cleansing and enrichment.

Responsibilities include:

* Data quality validation
* Timestamp standardization
* Stream-stream joins
* Stream-static joins
* Watermark handling
* Removal of invalid records

---

## Gold

The Gold layer prepares curated analytical datasets for reporting.

Responsibilities include:

* Business aggregations
* Reporting tables
* KPI preparation
* Analytics-ready datasets

# Execution Flow

The project executes in the following sequence:

1. Create the Unity Catalog landing volume.
2. Generate continuous streaming sensor data.
3. Execute the Bronze Delta Live Tables pipeline.
4. Execute the Silver Delta Live Tables pipeline.
5. Execute the Gold Delta Live Tables pipeline.
6. Publish curated data into Microsoft Fabric.
7. Build the semantic model.
8. Create Power BI reports.

# Databricks Execution Environment

This repository uses two different Databricks execution environments.

The following notebooks execute as standard Databricks notebooks:

* 00_setup_landing_volume_structure.ipynb
* 01_landing_generate_stream.ipynb

The following notebooks execute as part of a configured **Delta Live Tables (DLT)** pipeline:

* 02_dlt_bronze_layer.ipynb
* 03_dlt_silver_layer.ipynb
* 04_dlt_gold_layer.ipynb

These notebooks are intended to run inside the DLT pipeline runtime rather than as standalone notebooks attached to an interactive cluster.

When executed outside of a Delta Live Tables pipeline, the following error is expected:

```
ModuleNotFoundError: No module named 'dlt'
```

This is expected behavior because the `dlt` module is provided by the Delta Live Tables runtime and is not installed as a standard Python package.
